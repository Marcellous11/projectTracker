- Phase 12: Stream-state detection. Daemon (`scripts/session-watcher.js`) now tails each session's most recent JSONL (`~/.claude/projects/<folder>/<latest>.jsonl`, last 16 KB) and classifies a `streamState` per session: **tool** (last assistant event has `stop_reason: tool_use` with no following `tool_result`, OR a `user/tool_result` with no following assistant — Claude is mid-Bash/Read/etc.), **responding** (last write < 5 s ago AND last event is assistant/end_turn or user-non-tool_result — response just landed or about to land), **waiting** (default). Metadata events (`ai-title`, `agent-name`, `attachment`, `permission-mode`, `last-prompt`, `file-history-snapshot`) are filtered out of the classification — they fire on hooks and would mask the real signal. Daemon tick dropped from 3 s → 1.5 s so the state transitions feel real-time; dashboard auto-refresh dropped from 5 s → 3 s to match. `lib/sessions.js#getSessionPulseMap()` and `lib/session-daemon-state.js` pass through `streamState` + `lastEventAt`. New `@keyframes hud-pulse-tool` in `app/globals.css` mirrors the dramatic green pulse but in `--warm`/amber, paired with a `.hud-pulse-tool` class. Sidebar `PulseDot` precedence: tool (pulsing amber) → live/responding (pulsing green) → idle (solid green) → recent (blue) → cold (dim). Topbar gains a `TOOL` counter alongside LIVE/WAIT, hidden when zero; the dot itself pulses amber. Bonus: `devIndicators: false` in `next.config.mjs` hides Next's bottom-left N-circle that was overlapping the sidebar footer (`// SHOWING N · live · idle` counters). Verified live: PPS session detected as `streamState=tool` while the user has a Bash call in flight; topbar reads `LIVE 01 · TOOL 01 · WAIT 02`; tooltip on the PPS row reads `Claude is running a tool call · pid 47598`. - Phase 11: Dashboard auto-refresh. Phase 10's daemon proved real-time accurate (snapshot updates every 3s) but the dashboard never re-fetched — Next.js Router Cache held the rendered RSC payload until manual navigation or hard reload, so killing a `claude` process didn't visibly drop its sidebar pulse and starting a new one didn't light up. New `<AutoRefresh />` (`components/hud/auto-refresh.jsx`) is a "use client" ticker mounted once in `app/layout.jsx`: `setInterval` 5s (default; override via `NEXT_PUBLIC_DASHBOARD_REFRESH_MS`) calls `router.refresh()` to invalidate Router Cache and re-stream RSC; `document.visibilityState` gate pauses while tab hidden; `visibilitychange` listener snap-refreshes immediately on focus regain so coming back to the dashboard shows fresh state; error backoff doubles the interval up to 30s on consecutive failures, resets on success. Companion `<SyncIndicator />` (`components/hud/sync-indicator.jsx`) shows `// sync Ns` in the topbar right cluster — proves to the user the refresh is working. The two share `lib/refresh-store.js` (~15-line module-level pub/sub) so the indicator reads the last refresh time without React context or prop-drilling. Confirmed live: dev server logs show 2 GET / hits in 12s from the user's open tab (steady ~5s cadence), and `// sync 0s` markup renders in the topbar HTML. Trade-offs: each refresh re-renders the whole RSC tree (server-side cost amortized by daemon's 3s cache TTL on the snapshot read; client doesn't fetch anything itself). If pulse-only updates need to feel snappier later, swap to a dedicated `/api/pulse` poll + targeted client state. (2026-05-28)
- Phase 10: Real-time Claude session daemon. New host-side Node script `scripts/session-watcher.js` (~120 lines, no deps) ticks every 3s: `ps -ax -o pid,command` → filter rows where command is exactly `claude` (excludes Claude.app desktop helpers) → `lsof -p <pid> -a -d cwd -Fpn` per PID → `ps -p <pid> -o lstart=` for start time → encodes each cwd via the same logic as `lib/sessions.js#encodeCwd` (copy-pasted to keep the daemon Next-free) → atomic write to `data/live-sessions.json` (`.tmp` + rename so SSR never reads a half-written file). SIGTERM/SIGINT writes a final empty snapshot with `shutdownAt` so the dashboard knows the offline state is intentional. New `lib/session-daemon-state.js` is a React `cache()`-wrapped reader: returns `{online, ageMs, ts, sessions, byFolder}` with 30s staleness threshold; oldest-PID-wins when multiple processes share a cwd; graceful fallback to all-empty on missing/parse errors. `lib/sessions.js#getSessionPulseMap()` merges in the daemon byFolder so each pulse entry gains `processOpen`, `idle` (= processOpen && !liveNow), `pid`, `processStartedAt`. The map also surfaces daemon-known folders that have no JSONL directory yet (brand-new session before first write). Sidebar `PulseDot` (`components/sidebar.jsx`) now has 4 visual states: pulsing green (liveNow), solid green (idle), blue (recent), dim (cold); tooltip on idle shows `pid <n> · <age>`. Sidebar footer split into separate `N live` / `N idle` counters. Topbar (`components/topbar.jsx`): `LIVE` chip keeps mtime semantics; new dim-green `WAIT` chip counts idle (hidden when 0); `⊘ daemon` indicator shown when daemon snapshot is missing/stale. The Docker container can't see host processes, so the daemon lives on the host and writes to the already-bind-mounted `./data` directory; `data/live-sessions.json` is the cross-boundary signal. Auto-start ships as `launchd/com.mcurtis.project-tracker-watcher.plist` template — install snippet (`cp + sed __REPO__ → $(pwd) + launchctl bootstrap`) in README "Daemon" section.
